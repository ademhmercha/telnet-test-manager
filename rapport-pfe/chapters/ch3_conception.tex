\chapter{Conception}

\section*{Introduction}
Ce chapitre expose les choix architecturaux et les modèles de conception
retenus pour le développement de Telnet Test Manager. Nous présentons
successivement l'architecture générale de l'application, les diagrammes de
classes et de séquence des cas d'utilisation principaux, puis le modèle de
données MongoDB.

\subsection{Clean Architecture}

Le backend est structuré selon les principes de la
\textbf{Clean Architecture} \cite{martin2017clean}, garantissant l'indépendance
des règles métier vis-à-vis des détails d'infrastructure. La règle fondamentale
est que les dépendances ne peuvent pointer que \textit{vers l'intérieur}~:

\subsection{Justification du choix NoSQL}

MongoDB a été retenu comme système de gestion de base de données pour
plusieurs raisons~:
\begin{itemize}
  \item La nature \textbf{schéma flexible} des données de tests (étapes
        et logs de taille variable selon le type de test)~;
  \item La performance des opérations en \textbf{lecture/écriture intensive}
        sur les journaux d'audit~;
  \item La sérialisation native en \textbf{BSON/JSON} alignée avec
        l'écosystème Node.js~;
  \item La facilité d'intégration avec \textbf{Mongoose} (ODM) qui apporte
        la validation de schéma et les middlewares.
\end{itemize}

\section{Conception des interfaces utilisateur}
% -----------------------------------------------------------------

\subsection{Tableau de bord principal (Dashboard)}

Le tableau de bord offre un espace unique de sélection et d'exécution des
tests. Il est composé de~:
\begin{itemize}
  \item Un panneau de sélection hiérarchique (Poste~$\to$ Produit~$\to$
        Slot) permettant d'identifier l'équipement cible~;
  \item Un sélecteur de commandes Telnet avec prévisualisation des étapes~;
  \item Un terminal de résultats en temps réel (mise à jour par WebSocket)
        affichant chaque étape avec son statut et son horodatage~;
  \item Des boutons d'action~: Lancer, Arrêter, Exporter.
\end{itemize}

\subsection{Module de configuration}

L'interface de configuration adopte une présentation en \textbf{tableaux
imbriqués} avec des formulaires modaux pour les opérations CRUD. La
hiérarchie Poste~$\to$ Produit~$\to$ Slot~$\to$ Référence est navigable
via des onglets.

\subsection{Panneau d'administration}

Le panneau d'administration comprend cinq onglets~:
\begin{enumerate}
  \item \textbf{Vue d'ensemble}~: métriques globales (utilisateurs actifs,
        tests du jour, taux de succès)~;
  \item \textbf{Gestion des utilisateurs}~: tableau \gls{crud} avec
        assignation de rôles et permissions~;
  \item \textbf{Historique des tests}~: tous les tests de tous les
        utilisateurs, filtrables~;
  \item \textbf{Journal d'audit}~: liste chronologique des actions avec
        filtrage par utilisateur, action et date~;
  \item \textbf{Analytiques}~: graphiques de répartition des tests par
        statut, utilisateur et période.
\end{enumerate}

\section*{Conclusion}

Ce chapitre a posé les bases architecturales et conceptuelles du projet.
L'architecture N-tiers combinée aux principes de la Clean Architecture
garantit la séparation des responsabilités et la testabilité du code.
Les diagrammes de classes et de séquence documentent les interactions
entre les composants, tandis que les schémas MongoDB formalisent le
modèle de persistance. Le chapitre suivant détaille la mise en \oe{}uvre
technique de ces choix de conception.
