\section*{Introduction}
Le volet DevOps de ce projet est aussi ambitieux que le développement
applicatif lui-même. Ce chapitre couvre l'ensemble de la chaîne de livraison
logicielle~: de la conteneurisation Docker à l'orchestration Kubernetes/Helm,
en passant par le pipeline GitHub Actions et la supervision Prometheus/Grafana.

% -----------------------------------------------------------------
\section{Introduction au DevOps et au CI/CD}
% -----------------------------------------------------------------

Le terme \textbf{DevOps} désigne une culture et un ensemble de pratiques qui
rapprochent les équipes de développement (Dev) et d'exploitation (Ops) afin
d'accélérer la livraison de logiciels de qualité. Les trois piliers sont~:

\begin{itemize}
  \item \textbf{L'intégration continue (CI)}~: chaque poussée de code
        déclenche automatiquement la vérification, la compilation et les
        tests du projet~;
  \item \textbf{La livraison continue (CD)}~: le code validé est packagé
        et livré automatiquement sur l'environnement cible~;
  \item \textbf{Le monitoring}~: l'application déployée est surveillée en
        permanence pour détecter anomalies et dégradations de performance.
\end{itemize}

\subsection{Dockerfile du backend}

\begin{lstlisting}[language=docker,
  caption={backend/Dockerfile — image Node.js 20 Alpine}]
# Image de base légère (Alpine Linux ~50 MB)
FROM node:20-alpine

WORKDIR /app

# Copie des manifestes de dépendances (optimise le cache Docker)
COPY package*.json ./

# Installation des dépendances de production uniquement
RUN npm ci --only=production

# Paramètre de cache-busting (force la re-copie du code)
ARG CACHEBUST=1

# Copie du code source
COPY . .

EXPOSE 3002

CMD ["node", "src/main/server.js"]
\end{lstlisting}

\subsection{Dockerfile du frontend (multi-stage)}

\begin{lstlisting}[language=docker,
  caption={frontend/Dockerfile — build multi-stage Node.js + Nginx}]
# ─── Étape 1 : Construction de l'application React ───────────────
FROM node:18-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .

# Build React optimisé pour la production
RUN npm run build

# ─── Étape 2 : Serveur de production Nginx ───────────────────────
FROM nginx:alpine

# Copie des fichiers statiques construits
COPY --from=builder /app/build /usr/share/nginx/html

# Configuration Nginx avec reverse proxy
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
\end{lstlisting}

\begin{notebox}[title={Avantage du build multi-stage}]
  L'image finale du frontend ne contient que Nginx et les fichiers statiques
  compilés, sans les outils de build Node.js. La taille passe de~$\sim$900~MB
  (avec les dépendances de build) à~$\sim$25~MB (image de production).
\end{notebox}

\subsection{Stratégie de déploiement Rolling Update}

Kubernetes applique par défaut une stratégie de \textbf{rolling update}~:
les nouveaux pods sont lancés progressivement avant que les anciens soient
arrêtés, garantissant zéro interruption de service lors des mises à jour.
Paramétrage retenu~:

\begin{lstlisting}[language=yaml,
  caption={Stratégie de mise à jour du Deployment backend}]
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0    # Aucun pod indisponible durant la mise à jour
    maxSurge:       1    # Un pod supplémentaire temporaire autorisé
\end{lstlisting}

\subsection{Vue d'ensemble du workflow}

\begin{lstlisting}[language=yaml,
  caption={.github/workflows/ci-cd.yaml — pipeline CI/CD complet}]
name: CI/CD Pipeline — Telnet Test Manager

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

# Annule les exécutions précédentes du même pipeline pour la même branche
concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: true

jobs:
  # ─────────────────────────────────────────────────────────────
  # JOB 1 : Intégration continue Backend
  # ─────────────────────────────────────────────────────────────
  ci-backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js 20
        uses: actions/setup-node@v4
        with: { node-version: '20' }

      - name: Installer les dépendances
        run: cd backend && npm ci

      - name: Audit de sécurité (npm audit)
        run: cd backend && npm audit --audit-level=critical

      - name: Exécuter les tests
        run: cd backend && npm test --if-present

  # ─────────────────────────────────────────────────────────────
  # JOB 2 : Intégration continue Frontend
  # ─────────────────────────────────────────────────────────────
  ci-frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js 20
        uses: actions/setup-node@v4
        with: { node-version: '20' }

      - name: Installer les dépendances
        run: cd frontend && npm ci

      - name: Audit de sécurité (npm audit)
        run: cd frontend && npm audit --audit-level=critical

      - name: Build React (production)
        run: cd frontend && npm run build
        env: { CI: false }

  # ─────────────────────────────────────────────────────────────
  # JOB 3 : Déploiement (self-hosted runner, branche main uniquement)
  # ─────────────────────────────────────────────────────────────
  deploy:
    runs-on: self-hosted
    needs: [ci-backend, ci-frontend]
    if: github.ref == 'refs/heads/main'
    environment:
      name: production

    steps:
      - uses: actions/checkout@v4

      - name: Démarrer Minikube si nécessaire
        run: |
          minikube status || minikube start --driver=docker

      - name: Configurer kubectl
        run: kubectl config use-context minikube

      - name: Construire les images Docker (daemon Minikube)
        run: |
          eval $(minikube docker-env)
          docker build -t telnet-backend:latest  ./backend
          docker build -t telnet-frontend:latest ./frontend

      - name: Déployer avec Helm (atomic)
        run: |
          helm upgrade --install telnet-app ./helm/telnet-app \
            --create-namespace \
            --namespace telnet-app \
            --set secrets.mongoRootUsername=admin \
            --set secrets.mongoRootPassword=${{ secrets.MONGO_PASSWORD }} \
            --set secrets.jwtSecret=${{ secrets.JWT_SECRET }} \
            --atomic \
            --timeout 600s

      - name: Attendre la disponibilité des pods
        run: |
          kubectl rollout status deployment/telnet-app-backend  \
            -n telnet-app --timeout=240s
          kubectl rollout status deployment/telnet-app-frontend \
            -n telnet-app --timeout=180s

      - name: Résumé du déploiement
        run: |
          kubectl get pods -n telnet-app
          kubectl get services -n telnet-app
\end{lstlisting}

\subsection{Configuration du runner auto-hébergé}

Le job \texttt{deploy} s'exécute sur un runner \textbf{auto-hébergé}
(\textit{self-hosted}) installé directement sur la machine Windows locale
hébergeant Minikube. Cette configuration est nécessaire car~:
\begin{itemize}
  \item L'accès au cluster Minikube est local et non exposé sur Internet~;
  \item Les images Docker doivent être construites dans le daemon Docker
        de Minikube (sans registry externe)~;
  \item Les secrets d'infrastructure restent dans l'environnement local.
\end{itemize}

Le runner est enregistré sur GitHub via~:
\begin{lstlisting}[language=yaml,caption={Configuration du runner self-hosted}]
# Depuis le dossier d'installation du runner GitHub Actions
./config.cmd --url https://github.com/[ORGANISATION]/[REPO] \
             --token [TOKEN_RUNNER] \
             --labels self-hosted,windows,minikube
./run.cmd   # Démarrage du runner en écoute
\end{lstlisting}

const client = require('prom-client');

  name:       'http_requests_total',

  labelNames: ['method', 'route', 'status'],

  name:       'http_request_duration_seconds',
  help:       'Durée des requêtes HTTP en secondes',

const telnetTestsTotal = new client.Counter({
  name: 'telnet_tests_launched_total',
  help: 'Nombre total de tests Telnet lancés',

const wsConnections = new client.Gauge({
  name: 'websocket_active_connections',
  help: 'Connexions WebSocket actives',
});

module.exports = { httpRequestsTotal, httpDuration,
                   telnetTestsTotal, wsConnections };
